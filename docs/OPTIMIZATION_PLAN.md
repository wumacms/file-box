# Image Box 性能与架构优化方案

## 1. 核心架构与职责解耦 (Architecture Refactoring)

### 问题现状
- **Pinia Store 臃肿**：`imageBox.ts`（长达 300 多行）混合了状态维护、OSS SDK 调用、Supabase 数据库同步及 `localStorage` 缓存逻辑，违反了单一职责原则。
- **`App.vue` 职责过重**：作为根组件，它不仅通过 `v-if` 手写了简陋的路由逻辑，还塞入了侧边栏拖拽调整大小、响应式探测以及系统设置弹窗等大量本可以独立的模块。

### 优化方案
- **引入 Service 层**：剥离具体的 API 调用逻辑，新建 `src/services/ossService.ts` 与 `src/services/dbService.ts`，Store 仅负责状态的分发与业务流的编排。
- **引入标准路由**：使用 `vue-router` 替代当前的条件渲染，明确划分 `/login` 和 `/app` 页面环境。
- **UI 组件重构**：从 `App.vue` 剥离 `<Sidebar />` 与 `<SettingsDialog />` 组件，提取拖拽调整宽度的逻辑为 Composable（如 `useResizable.ts`）。

---

## 2. 数据一致性与“双写”问题 (Data Consistency)

### 问题现状
- **非原子性操作风险**：在上传、移动或删除时，前端都是先调用 OSS API 再调用 Supabase API。如果任何一步因网络中断而失败，就会产生“幽灵数据”（如 OSS 已删除但 DB 仍保留，或相反），系统缺乏回滚和补偿机制。
- **低效的循环写入**：`moveSelectedImages` 在批量移动时使用 `for` 循环逐条对 Supabase 发起 `update` 请求，当文件量较大时将产生严重的网络拥堵与性能瓶颈。

### 优化方案
- **数据库批量操作**：将多条数据的独立更新替换为单次基于 Supabase 的 RPC (Remote Procedure Call) 调用或批量 `upsert` 操作。
- **可靠的异步同步 (长效方案)**：尽可能降低前端对数据一致性的兜底压力。如果是私有部署，可利用存储服务自身的回调（Webhook）；若受限于前端直接操作，需加入严谨的 `try-catch` 补偿机制（失败时给予明确告警并终止后续链式调用）。

## 3. 性能与海量数据处理 (Performance)

### 问题现状
- **分页逻辑缺失**：`OSSProvider.ts` 的 `list` 方法硬编码了 `'max-keys': 1000` 且未处理分页令牌 `nextMarker`。一旦单目录下超过 1000 张图，超出的部分直接“消失”。
- **DOM 渲染灾难**：在 `ImageGrid.vue` 中如果真有上千张图片，直接通过 `v-for` 生成 DOM 节点会造成极大的内存占用与滚动的严重卡顿。

### 优化方案
- **完善 OSS 分页处理**：在 Provider 层支持并对外暴露 `marker` 参数，允许前端分批次获取目录下对象。
- **引入无限滚动或虚拟列表**：在 UI 层（如基于 Element Plus 或 VueUse）为图片网格增加触底加载（Infinite Scroll）机制，将 DOM 节点维持在健康水平。

---

## 4. 多用户安全与隔离漏洞 (Security)

### 问题现状
- **基础目录越权隐患**：在 `FolderTree.vue` 与移动文件对话框中，树节点的根级硬编码指向了 `path: ''`。如果未来系统多个用户处于同一个 OSS Bucket 内通过前缀隔离，查询空路径将直接暴露所有用户的文件。
- **数据库查询越权隐患**：检索 `searchImages` 目前仅为模糊查询全表，若项目对 Supabase RLS（行级安全）配置疏漏，极易变成全站搜索。

### 优化方案
- **确立前端沙箱边界**：一切根路径的起始应当被锁定在 `${user.id}/` 级别，不论是导航树还是目标移动选择器，彻底禁止向上跨越（上卷）。
- **强制 RLS 审查**：务必确认 Supabase 中 `images` 与 `user_settings` 表严格配置了行级安全策略 (`user_id = auth.uid()`)，用底层防御阻断任何越权读取可能。

---

## 5. 用户体验细节提升 (UI/UX)

### 问题现状
- **上传缺乏细节感知**：由于未使用 `OSSProvider` 的 `onProgress` 回调，用户上传 100MB 级别资源时只能面对僵死的 Loading 态，无法感知确切进度。
- **错误捕获不佳**：部分重要流程的异常被直接丢给 `console.error`，导致用户操作无效且不知所措。
- **搜索上下文丢失**：检索完成后不仅清空了目录列表，而且取消检索后无法无缝返回到之前的目录现场。

### 优化方案
- **可视化上传队列**：实现全局的任务上传队列面板，接入进度条。
- **全局拦截器交互**：所有核心的增删改查错误，均需向上抛出并通过 `ElMessage` 转换为用户友好的文案提示。
- **搜索与导航解耦**：将“搜索结果页”与“常规目录树”作为两种视图模式进行管理，不要简单地复写彼此的数据。
